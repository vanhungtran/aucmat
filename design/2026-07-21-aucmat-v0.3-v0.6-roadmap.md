# aucmat Roadmap: v0.3 through v0.6

Date: 2026-07-21
Status: Approved roadmap

## Purpose

This roadmap extends the v0.2 statistical-inference and multivariate-simulation foundation through four bounded releases:

| Release | Theme | Primary outcome |
|---|---|---|
| v0.3 | Multivariable biomarker panels | Binary |
| v0.4 | Multiclass AUC | Multiclass |
| v0.5 | Time-dependent AUC | Survival and competing risks |
| v0.6 | High-dimensional infrastructure | All supported outcomes |

Each release receives its own detailed design, implementation plan, and acceptance gate. Work on a later release must not weaken or bypass the statistical contracts established by earlier releases.

## Cross-Release Principles

- Preserve the v0.2 outcome, estimand, resampling, RNG, condition, and result-object conventions.
- Separate descriptive screening from confirmatory validation.
- Record the exact observations used by every estimate and comparison.
- Perform all data-dependent preprocessing, selection, and tuning inside the relevant training or resampling split.
- Name every AUC estimand explicitly; binary, multiclass, and time-dependent AUCs are not interchangeable.
- Keep optional ecosystem integrations behind adapters and `Suggests` dependencies where practical.
- Extend simulation alongside analysis so each new estimator has controlled validation data.
- Require numerical-oracle, simulation, edge-case, reproducibility, and package-check evidence before release.

## v0.3: Multivariable Biomarker Panels

### Goal

Move from univariate biomarker screening to honest construction and validation of combined biomarker scores for binary outcomes. `aucmat` will not become a general machine-learning framework; it will provide a focused panel workflow with strong leakage prevention and AUC inference.

### Public API

```r
fit_auc_panel()
validate_auc_panel()
compare_auc_panels()
plot_auc_panel()
simulate_auc_panel()
```

### Supported models

- Prespecified linear scores.
- Standard logistic regression.
- Ridge logistic regression.
- Lasso logistic regression.
- Elastic-net logistic regression.
- User-supplied fit and predict functions satisfying a documented contract.

Penalized fitting should use an established engine such as `glmnet` through an adapter rather than reimplementing coordinate descent.

### Training and validation

- External holdout validation.
- Stratified v-fold and repeated cross-validation.
- Nested cross-validation when tuning, feature selection, or preprocessing is data-dependent.
- Out-of-fold predictions stored with subject and split identities.
- Bootstrap optimism correction for prespecified workflows.
- Grouped or site-aware splits when subject clusters or sites are supplied.

Nested resampling follows the analysis/assessment separation used by `rsample`; the package may accept `rsample` objects through an optional adapter rather than building a competing split framework.

All filtering, normalization, imputation, biomarker selection, and hyperparameter tuning occurs inside the training portion of every split. A workflow that attempts to reuse full-data selected features in assessment data receives a leakage error or explicit exploratory status.

### Performance measures

- Raw and direction-prespecified ROC AUC.
- Partial AUC where requested.
- Calibration intercept and slope.
- Brier score and log loss.
- Sensitivity, specificity, PPV, and NPV at prespecified thresholds.
- Decision thresholds learned in training data and evaluated only in assessment data.

### Inference

- Confirmatory paired DeLong or bootstrap comparisons are supported on an untouched external validation cohort.
- Cross-validation estimates include between-split variability and repeated-resampling summaries.
- Pooled out-of-fold predictions are not treated as ordinary independent predictions for naive confirmatory DeLong inference.
- Panel-versus-single-biomarker comparisons use identical assessment subjects and prespecified score directions.
- Model-selection uncertainty is stated explicitly.

### Simulation

`simulate_auc_panel()` extends v0.2 with:

- Sparse and dense true coefficient patterns.
- Redundant correlation blocks.
- Null biomarkers.
- Interactions and nonlinear effects.
- Batch/site effects.
- A known oracle score when the data-generating model defines one.

### Result objects and plots

The panel object stores the fitted workflow, preprocessing steps, selected features, tuning results, resampling indices, out-of-fold predictions, external predictions, performance summaries, and provenance. Plots include coefficient paths, selection frequency, resampled AUC distributions, calibration, ROC curves, and panel-versus-marker differences.

### v0.3 acceptance gate

1. No preprocessing or feature selection can cross a split boundary.
2. External-validation comparisons agree with v0.2 inference for fixed predictions.
3. Nested and repeated resampling are reproducible and preserve subject grouping.
4. Penalized-model results agree with the selected external engine on fixed fixtures.
5. Null simulations do not show systematic optimism after nested validation.
6. Panel simulations recover expected ranking and calibration within Monte Carlo tolerances.
7. Documentation distinguishes model development, internal validation, and external validation.

## v0.4: Multiclass AUC

### Goal

Screen biomarkers and evaluate class-probability models when the outcome has more than two classes, without hiding materially different multiclass estimands behind one value.

### Public API

```r
aucmat_multiclass()
compare_auc_multiclass()
simulate_auc_multiclass()
plot_auc_multiclass()
```

### Estimands

- One-versus-rest AUC for every class.
- One-versus-one AUC for every unordered class pair.
- Hand-Till multiclass AUC.
- Unweighted macro and prevalence-weighted macro AUC.
- Micro AUC only for class-specific probability predictions where stacking has a clear interpretation.

Each result records class levels, score columns, averaging scheme, weights, and direction convention. A single continuous biomarker and a matrix of class probabilities use separate validated pathways.

### Inference

- Subject-level stratified bootstrap intervals.
- Bootstrap comparisons between biomarkers or models on common subjects.
- Joint resampling across every class contrast.
- Multiplicity adjustment across features, class pairs, or prespecified families.
- Global tests across class-specific AUC vectors.
- Explicit non-testable statuses for missing classes within a resample.

`pROC` provides Hand-Till point estimation but does not currently provide multiclass confidence intervals or comparison tests. v0.4 focuses its added value on matrix-wide inference, common-subject comparison, and simulation validation. Output naming should be compatible with the Hand-Till, macro, and weighted-macro terminology used by `yardstick`.

### Simulation

`simulate_auc_multiclass()` supports:

- User class prevalences.
- Gaussian-mixture and multinomial-probit generators.
- Correlated biomarkers.
- Target one-versus-one class AUCs by biomarker.
- Optional class-probability oracle scores.
- Class imbalance, label noise, batch effects, and overlapping classes.
- Feasibility diagnostics for incompatible pairwise class-separation targets.

Requested pairwise AUC matrices must satisfy direction and transitivity constraints implied by the chosen generative model. Nearest-feasible adjustment reports requested, implied, used, and achieved targets.

### Plots

- Biomarker-by-class AUC heatmaps.
- Pairwise class-AUC matrices.
- Faceted one-versus-rest ROC curves.
- Multiclass aggregate comparisons with bootstrap intervals.
- Class prevalence and missing-class diagnostics.

### v0.4 acceptance gate

1. Point estimates agree with independent Hand-Till and one-versus-rest fixtures.
2. Results are invariant to harmless class reordering after explicit score-column alignment.
3. Bootstrap inference retains the complete subject-level class structure.
4. Null multiclass comparisons meet Monte Carlo Type I error tolerances.
5. Simulation detects incompatible target matrices and validates feasible targets.
6. Macro, weighted-macro, micro, and Hand-Till values cannot be mislabeled or silently substituted.

## v0.5: Survival and Competing-Risk AUC

### Goal

Extend biomarker-matrix screening to censored time-to-event outcomes using explicit time-dependent ROC definitions.

### Public API

```r
aucmat_survival()
compare_auc_survival()
simulate_auc_survival()
plot_auc_time()
```

### Initial estimands

- Cumulative/dynamic AUC at prespecified horizons.
- Integrated AUC over a prespecified time interval.
- Competing-risk AUC for an explicit cause.
- Explicit competing-risk control definition.

Incident/dynamic and other time-dependent definitions are deferred until they have separate estimand objects and validation. The package never labels a generic result merely as "survival AUC".

### Estimation and inference

- Inverse-probability-of-censoring weighting.
- Documented censoring-weight model and diagnostics.
- Influence-function or bootstrap confidence intervals.
- Paired biomarker comparisons at every horizon.
- Multiplicity adjustment across time points and biomarker contrasts.
- AUC-difference curves with simultaneous or pointwise bands clearly identified.
- Integrated-AUC comparisons.
- Warnings for extreme weights, low risk-set sizes, and unsupported horizons.

The implementation aligns with the cumulative/dynamic and competing-risk definitions used by `timeROC`. Independent validation may use `timeROC` and `riskRegression`, which already provide time-dependent comparison and broader prediction-score machinery.

### Simulation

`simulate_auc_survival()` supports:

- Proportional-hazards and accelerated-failure-time models.
- Correlated biomarkers.
- Administrative and random censoring calibrated to a target fraction.
- Optional informative censoring as a documented stress test.
- Competing causes.
- Target AUCs at selected horizons when feasible.
- Requested-versus-achieved AUC trajectories and censoring diagnostics.

Multiple target horizons may be incompatible with a chosen parametric model. Feasibility is checked before sampling, and nearest-feasible mode reports the complete trajectory adjustment.

### Plots

- AUC versus time with intervals.
- Biomarker-by-time heatmaps.
- Difference curves between biomarkers or models.
- Censoring survival, weight, and risk-set diagnostics.

### v0.5 acceptance gate

1. Point estimates and paired comparisons agree with independent time-dependent-AUC fixtures.
2. No estimate is produced beyond its supported risk-set horizon.
3. Censoring weights and competing-risk control definitions are stored and displayed.
4. Time-point multiplicity and simultaneous-band semantics are explicit.
5. Null simulations meet Monte Carlo Type I error and coverage tolerances.
6. Survival simulations reproduce requested censoring and feasible AUC trajectories within stated tolerance.

## v0.6: High-Dimensional and Bioconductor Infrastructure

### Goal

Analyze omics-scale matrices without forcing all inputs into one dense in-memory base matrix, while producing numerically equivalent results across backends.

### Supported inputs

- Base matrix and data frame.
- Sparse `Matrix` objects where score semantics are valid.
- `DelayedMatrix` and `HDF5Array`.
- `SummarizedExperiment`, preserving assay, feature, and sample metadata.

The `SummarizedExperiment` adapter explicitly handles its feature-by-sample convention while `aucmat` internally uses subject-by-feature semantics. It validates sample alignment before analysis.

### Processing architecture

- Matrix-like adapter with dimension, name, block-read, and metadata methods.
- Feature-block processing without whole-object dense coercion.
- Reusable cached ranks for repeated resampling where memory permits.
- Reproducible serial and parallel backends.
- Optional `BiocParallel` integration.
- Checkpoint and restart for long validation or stability jobs.
- Progress, memory, failure, and backend diagnostics.
- Configurable block size and memory budget.

`DelayedArray` provides block processing for in-memory and on-disk array-like objects; `HDF5Array` supplies HDF5-backed dense and sparse containers. These should be adapters, not mandatory dependencies of the core CRAN installation.

### Statistical safeguards at scale

- Joint covariance and all-pairs operations remain explicitly bounded because storage can grow as `O(p^2)`.
- Global or pairwise inference requires a prespecified or screened subset under the post-selection rules established earlier.
- Chunking cannot alter missingness sets, factor direction, tie handling, seeds, or numerical results.
- Backend-specific approximation is forbidden unless named, optional, and validated.

### Benchmark matrix

Benchmarks cover tall, wide, sparse, delayed, and HDF5-backed inputs for screening, bootstrap comparison, stability, and simulation validation. They report elapsed time, peak resident memory, disk reads, temporary storage, and numerical deviation from the in-memory reference.

GPU and distributed-cluster execution remain outside v0.6 unless benchmarks demonstrate a concrete need and a stable R integration.

### v0.6 acceptance gate

1. All supported backends agree with in-memory reference results within documented tolerance.
2. Delayed and HDF5 inputs are never wholly coerced to dense matrices.
3. `SummarizedExperiment` sample and feature metadata round-trip correctly.
4. Parallel results satisfy the established reproducibility contract.
5. Interrupted checkpointable jobs resume without duplicating or skipping work.
6. Memory remains within the declared budget for representative on-disk workloads.
7. Core installation remains usable without optional Bioconductor dependencies.

## Dependency Sequence

```text
v0.2 statistical core and multivariate simulation
  -> v0.3 prediction panels and honest resampling
  -> v0.4 multiclass estimands and simulation
  -> v0.5 time-dependent estimands and simulation
  -> v0.6 backend abstraction across all supported outcomes
```

Scaling is scheduled last so the package first stabilizes the estimands and result contracts that every backend must reproduce. The v0.2-v0.5 internal engines must nevertheless avoid unnecessary dense coercion and isolate matrix access so v0.6 does not require a statistical rewrite.
